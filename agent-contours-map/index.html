Роли контуров и как они взаимодействуют

Каждый агент — отдельная сессия Claude Code с одной зоной ответственности. Контуры не пишут в чужие зоны: пересечение решается передачей задания владельцу, а необратимые действия проходят через независимого reviewer'а и оператора. Ниже — обезличенная модель ролей и потоков между ними.

Роли контуров

Правила взаимодействия

R1
Одна зона — один владелец

Контур правит только свою папку/домен. Смежные проекты не трогает даже при сходстве кода (например, два независимых firmware-проекта — взаимные правки запрещены).

R2
Пересечение → задание владельцу

Задача, затрагивающая чужую зону, не выполняется своими руками — даже read-only. Вместо этого формируется промпт-задание для контура-владельца. Никто не выходит за свою колонку.

R3
Reviewer судит, не исполняет

Независимый reviewer проверяет чужие артефакты по факту (actor≠verifier). Необратимое — push / merge / deploy / tag — только по явному «да» оператора. Вердикт ≠ исполнение.

R4
Вверх — только сбои

Контуры шлют reviewer'у по каналу координации лишь системные события (краш делегации, порча реестра, утечка, блокер). Рутинные шаги и статусы не эскалируются — экономия внимания.

R5
Делегирование вниз

Bulk-разбор, шаблонная генерация и суммаризация уходят на локальную модель и фоновых субагентов; контекст оператора и frontier-модель — для решений и синтеза.

R6
Тройная избыточность

Навыки и незаменимые артефакты дублируются в облачном бэкапе, локальном git и на GitHub — восстановление любого контура после сбоя гарантировано.
